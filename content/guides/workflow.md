本文档专门针对 **"LLM + Workflow"** 这一技术分支进行深度拆解。

> 💡 **核心概念澄清**：
> **Workflow（工作流）** 和 **Agentic（自主智能体）** 是两个截然不同的理念。
> *   **Workflow 的核心是"控制"**：流程的走向（先干什么，后干什么）是由**人类开发者**通过代码（`if-else`）或可视化界面的连线画死的。大模型在这个流程里只是一个"高级外包工人"，只在特定的节点上完成特定的任务（如翻译、提取关键词）。
> *   因此，Workflow 具有极高的**稳定性和确定性**，是目前大中小厂落地日常业务（如自动填表、工单审批、批量翻译、智能客服中控）最绝对的主力。

> 📚 **核心理论框架来源**：
> * **Wu, T., et al. (2022). "PromptChainer: Chaining Large Language Model Prompts through Visual Programming"**. (该论文最早系统性地提出了将大型任务拆解为多个独立 LLM 节点，并通过有向图进行串联的 Workflow 理论基础，是如今 Dify、Coze 等可视化平台的学术雏形)。
> * **Zhou, D., et al. (2022). "Least-to-Most Prompting Enables Complex Reasoning in Large Language Models"**. (奠定了为何必须将复杂业务解构为原子步骤的理论基石，证明了"分治法"能极大提升大模型输出结果的稳定性和精确度)。
> * **Malewicz, G., et al. (2010). "Pregel: A System for Large-Scale Graph Processing"**. (Google 提出的顶级图计算模型论文。这是目前 LangGraph 等硬核工作流编排框架在处理复杂图流转与状态传递时的底层计算机科学灵感来源)。

## 标准开发流程 (5步法)

### 第 1 步：业务 SOP 的极度拆解 (SOP Deconstruction)
*   **痛点**：如果你给大模型一个超长的 Prompt，既要求它分析财报，又要求它提取数据，还要求它翻译，大模型极容易崩溃、遗漏指令（这叫"既要又要"陷阱）。
*   **大厂做法**：将传统业务的 SOP（标准作业程序）翻译成流水线图纸。
    *   *错误做法*：用一个极其复杂的 LLM 节点企图搞定所有事。
    *   *正确做法（拆解）*：节点 1（专注判断语言类别） -> 节点 2（专注翻译成中文） -> 节点 3（专注从中文里提取金额数值）。
> 📚 **学术延伸**：这种做法在学术界被称为 **Prompt Chaining（提示词链）**。相比于让模型自由发散思考的 Chain of Thought (CoT)，Prompt Chaining 是一种由外部硬编码强制规定的确定性执行链。

### 第 2 步：原子节点开发 (Atomic Node Definition)
在代码或低代码平台层面，大厂通常会把工作流里的节点严格分为三类：
1.  **大模型节点 (LLM Node)**：黑盒输入输出。输入一段 Prompt 和变量，输出一段文本（最好要求输出 JSON 结构）。不涉及任何外部请求交互。
2.  **工具代码节点 (Code/Tool Node)**：**完全不调用大模型**。纯粹执行 Python/Node.js 等脚本代码，比如执行一段 SQL 查数据库、调取天气 API、发送企业微信通知。
3.  **路由/逻辑节点 (Router Node)**：执行传统的 `If/Else` 或 `Switch` 逻辑。比如根据上一步 LLM 提取的"投诉严重程度"，如果严重程度为"高"，则流转到"发送短信给主管"节点；如果为"低"，则流转到"普通排队"节点。

### 第 3 步：全局状态与黑板模式 (State & Blackboard Pattern)
*   **痛点**：节点与节点之间默认是隔绝的，节点 3 怎么知道节点 1 提取了什么信息？
*   **大厂做法**：引入一个在全图流转的 **State（状态对象 / 贯穿全局的 JSON 结构）**。
    *   把它想象成一个在流水线传送带上的"篮子"。
    *   节点 1 翻译好文本，把它放进篮子的 `translated_text` 字段里；篮子传到节点 2，节点 2 从该字段读取文本，提取出名字后，再把名字放到篮子的 `person_name` 字段里。
    *   这种设计保证了上下文数据的穿透，且各个节点互不干扰。
> 📚 **理论溯源**：这在传统 AI 架构与分布式计算中被称为 **Blackboard System (黑板架构)**，所有独立的"专家（执行节点）"都在一块共享的中心化黑板上读取和更新信息。

### 第 4 步：硬编码的异常捕获与容错 (Hard-coded Fallback)
*   **痛点**：大模型 API 请求超时，或者你让它严格输出 JSON，它却多嘴说了一句"好的，以下是你要的 JSON："导致后续解析代码瞬间崩溃。
*   **大厂做法**：在 Workflow 中，容错完全依赖**人类预设的死规则**（这与 Agentic 遇到报错会让大模型自己看报错信息去反思修改截然不同）。
    *   **重试机制 (Retry)**：配置失败后等待 2 秒重试，最多重试 3 次。
    *   **兜底降级 (Fallback 支线)**：如果大模型彻底抽风解析失败，系统会直接将其分发到一条名为"人工处理异常池"的默认支线上，防止整个业务流卡死瘫痪。

### 第 5 步：可观测性埋点 (Observability & Tracing)
*   **做法**：在每一个节点进出的地方打上日志（Log）和链路追踪点（Trace）。
*   在企业级应用中，一旦客户反馈"昨天的报表少了一部分"，研发必须能在监控平台上清晰地看到：数据经过节点 2 时，消耗了 1500 个 Token，耗时 3.2 秒，并且能完整查看大模型在那个特定时刻到底被喂入了什么 Prompt，以及它的原始输出是什么。

## 落地框架与平台选型

1.  **低代码/零代码平台可视化拖拽（适合中小厂、传统企业转型、或快速 MVP 验证）**：
    *   **代表工具**：`Dify`, `Coze (扣子)`, `FastGPT`。
    *   **优势**：所见即所得，画完流程图点一下按钮，直接生成后端 API 接口，极大降低了非 AI 开发人员的门槛。
2.  **纯代码硬核编排（适合一线大厂、复杂并发业务线、核心后端研发）**：
    *   **代表框架**：`LangChain (LCEL 语法)`, `LangGraph`, `LlamaIndex (Query Pipelines)`。
    *   **优势**：支持极度复杂的业务逻辑、高并发队列处理、强绑定数据库本地事务，不受可视化拖拽平台现有组件的限制。
